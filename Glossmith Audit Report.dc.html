Spec gaps
Verdict

The app is a competent single-file prototype, but it is not architected around a single source of truth — and the two most important behaviours in the spec (never store balance, never overwrite payment history) are violated at the core. The backend does not calculate anything: it stores raw rows and overwrites the deposit field to fake a cleared balance, destroying the real payment record. Every derived value (balance, stage, "paid in full", cash-vs-M-Pesa split) is computed in the browser, so the frontend — not the backend — is the de-facto source of truth.

This confirms your instinct: it should be rebuilt, not patched. There is no Customers table, no Vehicles table, and no Payments table — the entire relational model the spec calls for is missing. Good news: the pricing catalogue, UI language, and Daraja scaffolding are salvageable and will carry into the rebuild.

A

Money & data integrity

The findings that corrupt the numbers or destroy history.

A1 · CRITICAL Data loss

"Clear balance" overwrites the deposit and erases what was actually paid

In setStage_ the only way to "clear" a balance is setCell_(sheet, idx, "deposit", finalAmount). The real deposit amount is destroyed, so after clearing you can no longer tell what was paid up front vs. paid at collection. There is no payment method, no transaction code, no timestamp. This directly violates the spec's "Clear Balance must NOT immediately set Balance to zero" and "Never overwrite payment history."

Fix → A real Payments table. Clearing = insert a payment record (method, amount, code, time). Total Paid = SUM of payments. Balance is never written.
A2 · CRITICAL No source of truth

Balance, stage and "paid" are calculated only in the browser

The backend's doGet returns raw sheet rows. Every business value — saleBalance(), deriveStage(), isCleared() — lives in index.html. The backend even has a different stage rule than the frontend, so the two disagree. This is the root cause of "inconsistent balances across screens": there are two brains.

Fix → A Business Layer in Apps Script computes totalPaid, balance, status, stage on read. Frontend renders, never derives.
A3 · CRITICAL Wrong numbers

Closing a job inflates "Total collected" & misattributes the payment mix

Because closing sets deposit = finalAmount, the dashboard's "Total collected" (a sum of deposit) jumps to the full amount even though only a partial payment actually changed hands. The cash-vs-M-Pesa split then attributes that entire inflated amount to the single paymentMethod field — so a cash deposit later cleared by M-Pesa is reported as 100% one method.

Fix → Per-payment method & amount. The split sums real payment rows, not a scalar.
A4 · CRITICAL Field mismatch

The frontend sends make, but the Sales sheet has no make column

submitSale() posts make: logForm.make, but SALES_HEADERS only defines model. addRow_ writes by column name, so make is silently dropped on every sale. This is exactly the "frontend payload doesn't match Apps Script fields" bug — and why vehicles lose their make. It's a template for a whole class of silent drops.

Fix → One shared schema constant, validated on write; unknown fields raise an error instead of vanishing.
A5 · CRITICAL Workflow inverted

A job can be closed with an outstanding balance — the rule is backwards

The spec: "A job CANNOT be closed if Balance > 0." The code does the opposite — moving to Closed force-clears the balance to zero (stage === "Closed" → deposit = finalAmount). So closing hides unpaid money instead of preventing it.

Fix → Server rejects Closed when balance > 0, returns the exact spec message.
A6 · CRITICAL Race condition

No locking — concurrent writes corrupt rows

Every mutation is read-then-write (findRowIndex_ → setCell_) with no LockService. Two staff acting at once (very likely on a shared shop tablet + phone) can interleave and write to the wrong row or clobber each other. repairAllSheets is worse — it clearContents() then rewrites, so a mid-run failure loses the sheet.

Fix → LockService.getScriptLock() around every write; append-only payments avoid most contention.
A7 · CRITICAL Security

Every mutating endpoint is unauthenticated

The PIN is only checked in the checkPin action to unlock the UI. doPost never re-verifies it, so anyone who knows the public /exec URL can addSale, deleteSale or clear balances with a single curl. The role is also stored in sessionStorage and trusted client-side for owner-only tabs.

Fix → Issue a signed session token on PIN unlock; require it on every write; enforce owner-only actions server-side.
B

Data model & relationships

The relational structure the spec asks for does not exist.

B1 · HIGH

No Customers or Vehicles tables — both are derived from Sales in the browser

vehiclesFromSales() rebuilds vehicles from sale rows on every load. If a sale has no plate, the vehicle is invisible (the "Vehicles page blank" bug). There is no place to store a vehicle's colour, year, or a customer's additional phone numbers — the spec's core entities have nowhere to live.

B2 · HIGH

Customer identity is guessed by fuzzy key, causing duplicates

ownerKey() groups by last-9-digits of phone, else lowercased name, else plate. A typo in a name or a number entered two ways creates a second "customer". There is no stable customer ID, so history splits silently — exactly the "duplicate customers caused by phone formatting" risk in the spec.

B3 · HIGH

Photos: only two phases, one flat Drive folder

The model has intakePhotos/handoverPhotos only — the spec needs Before / During / After. All files land in one shared Glossmith Job Photos folder, not a per-job folder, so they're unmanageable at volume.

B4 · HIGH

No technician-notes timeline

Only two free-text fields (intakeNote, handoverNote) exist. The spec's timeline (Received → Inspection → PPF Started → … each with date, time, user, message) has no table.

B5 · MEDIUM

Duplicate keys in the VEHICLES catalogue

MG, Bentley and Aston Martin each appear twice in the object literal; the later definition silently wins and the first is dead data. Trivial, but a sign the catalogue needs a single normalized source.

C

Backend & API

Structure, versioning, and the response contract.

C1 · HIGH

No layering — routing, business rules and sheet I/O are intermixed

The spec asks for Repository / Validation / Business / Response / Utility layers. Today doPost switches directly into functions that both validate lightly and hit the sheet. There is no validation layer and no consistent response envelope.

C2 · HIGH

Responses are inconsistent — no standard envelope

Handlers return ad-hoc shapes: {id, ok}, {ok, error}, {valid, owner}. The spec wants every response to carry success, message, version, timestamp, updatedObject. No endpoint returns the updated object, so the frontend must do a full reload after every action.

C3 · HIGH

Version check is a brittle string compare with a scary red banner

EXPECTED_SCRIPT_VERSION must exactly equal SCRIPT_VERSION. Any redeploy where they drift (a common mistake) throws the persistent red "talking to an old backend" warning users are seeing. There is no real health endpoint reporting deployment ID, sheet connection, or build number as the spec requires.

C4 · MEDIUM

Full-table reads on every GET

Each load returns all sales, promos, events, members, jobData and staff in one payload with no paging, filtering or caching. Fine at 50 rows; it degrades linearly and will feel slow on a phone at a few thousand.

C5 · MEDIUM

Errors are stringified and swallowed

The top-level catch returns {ok:false, error:String(err)} with no logging, no error code, no stack. The spec wants user-friendly messages + developer logging + retry.

D

Frontend, state & UX

One 1,130-line file; string-built HTML; full re-render per action.

D1 · HIGH

"Reload everything" after every write

Every action ends in await loadData(); renderAll(); — a full network round-trip plus a re-render of all eight panels. Because the backend returns no updated object, there's no optimistic update path. On a slow shop connection each tap feels laggy.

D2 · HIGH

HTML built by string concatenation with hand-rolled escaping

Every view is a giant template string. Escaping relies on remembering to call esc() at each interpolation, and onclick="fn('${id}')" handlers are wired by id string — fragile and an injection risk if any field slips through.

D3 · HIGH

Not mobile-first where it counts: top tab-strip, no bottom nav

Navigation is a horizontally-scrolling pill strip; the spec wants a bottom nav with large touch targets. Job actions use prompt()/confirm() (e.g. M-Pesa phone entry), which are jarring on phones.

D4 · MEDIUM

Balance logic duplicated & discount is a flat 8% regardless of the promo

Discount is always DISCOUNT_RATE = 0.08, ignoring each promo's own configured rate. The final amount is computed in three places (preview, submit, balance hint) that can drift.

D5 · MEDIUM

No offline support, no upload retry, no cache-busting

Photo upload has compression but no retry on failure; the app has no service worker / offline queue; and static assets on Vercel aren't versioned, so users can be served a stale bundle after a deploy — one of the symptoms you described.

D6 · MEDIUM

Receipts/PDFs open a blank window and write unbranded HTML

window.open("") + document.write is popup-blocker-prone and produces plain system-font documents with no Glossmith branding, no QR code, no photo grid by phase.

The rebuild

Target architecture

New Google Sheets schema (7 tables, fully relational)
Customers — id, name, primaryPhone, extraPhones[], createdAt
Vehicles — id, customerId, plate*, make, model, year, colour, class
Jobs — id, jobNo, vehicleId, customerId, services, amount, discount, finalAmount, stage, technician, channel, promoCode, createdAt
Payments — id, jobId, amount, method, mpesaCode, kind, receivedBy, at
Photos — id, jobId, phase(before|during|after), url, driveId, at
Notes — id, jobId, message, user, at (timeline)
Promos / Staff / MpesaLog — carried over, normalized
* plate unique & uppercased · phones normalized to +254 · balance never stored.
Backend layers (Apps Script)
  1. Repository — read/write by column name, LockService
  2. Validation — required, ranges, duplicates
  3. Business — balance, stage, totals (only here)
  4. Response — {success,message,version,timestamp,updatedObject}
  5. Utility — normalize phone/plate, IDs, dates
  6. Health — version, deployId, DB status
Frontend modules
  1. api — one fetch wrapper, retry, version guard
  2. state — single store, updates from updatedObject
  3. components — render helpers, no derivation
  4. screens — dashboard, pipeline, sale, vehicles…
  5. whatsapp — device detect + +254 + templates
  6. pdf — branded job card / receipt + QR
The one rule that fixes most bugs

Money lives in the Payments table as append-only rows. totalPaid = Σ payments, balance = max(0, finalAmount − totalPaid), computed once, on the server, returned on every read. Dashboard, pipeline, receipt, reports, customer & vehicle history all render the same server number. Clearing a balance and closing a job both become "insert a payment" — history is never overwritten, and the numbers can no longer disagree.
